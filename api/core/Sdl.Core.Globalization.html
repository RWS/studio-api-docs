    Language

    LanguageBase

    Represents a language in the sense of a System.Globalization.CultureInfo object, using a mechanism that allows basic manipulations and adds metadata not available in System.Globalization.CultureInfo such as UseBlankAsWordSeparator.

    This class was originally introduced as an alternative to the System.Globalization.CultureInfo class in the .NET Framework so that when working with files we could convert the file on a system that supported a particular language (i.e. theSystem.Globalization.CultureInfo for this language abbreviation was known), and still allow the file to subsequently be opened on a system that did not yet have support for this language.

    Depending on what processing is to be done this may be a perfectly valid use case, and it would have been too harsh a restriction to require support for the language to be installed on all systems that may ever look at the file.

    Now, we no longer require a System.Globalization.CultureInfo to be installed on a system for this kind of use, since instead of creating it direcly from .NET, a StableCultureInfo should instead be retrieved, which is built from metadata we control, such that the same set of languages is available on all platforms. A StableCultureInfo can be retrieved either from GetCultureInfoAsync() or GetStableCultureInfoAsync(String).

    Under normal circumstances the language should be retrieved via GetLanguageAsync(String, Boolean) or other LanguageRegistryApiAsync method.

    In addition, in the past this wrapper allowed a language to be represented by a plain string when no System.Globalization.CultureInfo for this language is supported on the system, and thus allowed the language to be represented until some truly language specific processing is needed (i.e. when the corresponding System.Globalization.CultureInfo is required). Such an object could be created with the Language(string) constructor. However, this is now deprecated, since all supported cultures can now be used on all platforms, as StableCultureInfo. Flows like that should be adjusted either just to use a string, or to use a CultureCode, or to use StableCultureInfo.

    When supported by our products the corresponding CultureInfo can be obtained directly from the System.Globalization.CultureInfo property. If not supported this will throw an UnsupportedLanguageException.

    Use the IsValid and GetIsSupportedAsync() properties to determine if the language has been set and whether it is supported by our products.
